    冴羽 2021/12/12

    # 整体流程

    连接器->分析器(词法、语法)->优化器(生成执行计划:用啥索引、join的表顺序)->执行器(操作引擎，返回结果)->存储引擎(存储数据，提供读写接口)

    # 专业名词定义

    crash-safe 数据异常重启，之前提交的数据库记录也不会丢失

    # 常用命令

    查询进程
SHOW FULL PROCESSLIST;

    # 常见参数

    • wait_timeout 默认值8小时, 连接空闲太久自动断开，如果mysql断开后，客户端发起，就会显示错误 Lost connection to MySQL server during query。，需要重新连接执行命令
    • innodb_flush_log_at_trx_commit 设置为1，每次事务提交都会讲redo log磁盘中
    • sync_binlog 设置为1，每次事务中binlog都会刷到磁盘中

    # 关联对比

    # redo log & binlog

    比较维度 binlog redo log
    层次 mysql server innode存储引擎
    文件类型 逻辑日志 二进制形式，有ROW、Statement、Mixed三种模式 物理日志，记录如何更新数据页
    作用 use for Point-in-time Recovery,用于主从同步、数据异构 handle crash recover,保证持久化
    写入时机 事务提交时写入 固定时间或者事务提交时(**没有提交的事务也会flush to disk，**后续如果事务回滚，采用undo log回滚数据)
    文件使用形式 循环覆盖 累加

    几个注意点

    • redo log是storage engine内部日志形式，binlog是mysql server维度给所有engine用的归档日志，引擎内部需要处理cash recover的问题，保证持久化，但是他不能反向依赖上一个维度的binlog, 因此只能基于redo log to handle cash recover，这里因此就引入了两个日志之间的consistency，也就带来two phase commit的问题。

    • tow phase commit的流程中，涉及到两方，协调方executor，参与方innode engine

      • prepare阶段

        • engine flush redo log, 设置为prepare状态
        • executor执行器flush binlog(有些疑惑在于这个步骤算在prepare还是commit其实好像都可以，放在commit应该是先执行)

      • commit阶段

        • commit transaction(设置redo-log为committed状态)

    innodb自带 mysql服务层实现 物理日志 具体某个数据页做什么修改 逻辑日志 具体语句的逻辑 循环复写/追加
